The data layer is essentially the interface between the website, or what happens on the website, and Google Tag Manager. The CMP places information into the data layer. Tag Manager can then reuse that information and notices when something happens there.

There are several important CMP variables. Our variables start with “cmp,” with a capital P in some places. One of them is “cmpGDPR.” This variable contains a 1 if GDPR applies to the user.

That works well for simple tags.

However, you need to be aware that the consent event can sometimes fire relatively late. Imagine the user arrives on the website. The layer is displayed. It may happen that the user first looks at the layer for three minutes and only then clicks Accept.

There may be certain tags where the sequence matters, or which should not fire on every page view but only on certain events. For example, imagine a shopping cart event. You have a tag that should fire whenever someone clicks “Add to cart” in your shop. The page does not reload; the user simply goes from three items in the cart to four items.

This event may happen before the consentmanager event. If you have a layer that does not cover the whole page, the user can still use the page without making a decision. So the user might first click “Add to cart” and only then click “Accept.” Or the other way around: the user may already have accepted and only then adds something to the cart.

You do not know the order. It is an event that can occur significantly later than the page load.

To catch such events, you can either pre-filter the event. For example, you could take the original “Add to cart” trigger and add a condition to it: fire only if the provider, for example Google Analytics with ID s26, has consent. But that only works if consent already exists at the moment the user clicks “Add to cart.”

The other case — where the user first clicks “Add to cart” and then gives consent — would not be covered.

That is why, in most cases, it is more practical to replace the original trigger with a trigger group.

Google Tag Manager has had so-called trigger groups for a while now. I create a new trigger and choose “Trigger Group.” As the name suggests, it is simply a group of different triggers.

The advantage is that I can add multiple triggers, for example “Add to cart” and “Analytics has consent.” A trigger group works so that Tag Manager fires the tag connected to this trigger only when both triggers have been activated. The order does not matter.

So it does not matter whether the user first clicked “Add to cart” and then accepted Analytics, or whether the user first accepted Analytics and then clicked “Add to cart.” As soon as both have occurred, the tag will fire.

I save that and connect it to my shopping cart Analytics event. In this example, Analytics would only count once consent exists and the user has clicked “Add to cart.”

That sounds nice, but there is one thing to be aware of: trigger groups only fire once. In this example, if the user clicks “Add to cart” five times and then clicks Accept once, the tag would still only fire once for that page view. Now we will move on to Google Consent Mode.

Many of you have probably already heard of Google Consent Mode. Google is also trying to push this topic.

What is Google Consent Mode? In simple terms, it is a mechanism originally designed for Google Analytics and Google Ads — the Google tracking products — to soften the tracking when there is no consent.

Without Consent Mode, there are basically only two legal variants: if there is consent, I activate Google Analytics and Google counts everything. If there is refusal, I block Google Analytics and Google Analytics counts nothing.

Google Consent Mode creates something in between. The idea is not that refusal means you are allowed to count nothing at all. Rather, personal data must be handled more sensitively. We will leave the legal aspects aside for a moment, but if you want to create a grey area between acceptance and rejection, this is where Google Consent Mode comes in.

As mentioned, it was designed for Google Ads, Floodlight, and Analytics. These are the products. It is also integrated into Google Tag Manager to simplify control.

When enabled, Analytics automatically detects that Consent Mode is on and that only basic information may be measured. For example, IP addresses are no longer processed in the same way; they are only used to determine the country. Browser information is reduced. Advertising cookies and similar advertising-related things are not set, and existing ones are not read.

So everything relating to ePrivacy and TTDSG legislation is reduced. Only when consent is given does consentmanager tell Google Analytics that consent now exists. Google then knows it can start processing the rest.

The legal risk remains. We went into that in yesterday’s webinar, so please watch that video if you want more detail. I will touch on it briefly again later. But the legal risk remains, and the implementation, especially in Google Tag Manager, is still a little shaky in some places. It is not yet where one would ideally like it to be.

That said, there are very good approaches. I have already mentioned the Google Tag Manager integrations. The ideas behind them are very good and creative. At the moment, however, there are still some limitations.

To use Consent Mode, the first thing you need to do is log in to consentmanager, go to your CMP settings, edit the CMP, and go to “Other settings.” At the bottom there is a checkbox for Google Consent Mode. Tick that box. This activates the logic on the consentmanager side, so consentmanager can send the consent signal.

It is not enough to simply tick the box in our system and then be done. These lines must be integrated into the website code. The lines make sure that Analytics, Google Ads, or Google Tag Manager have a default status. The default status for data processing is set to denied. That is what activates Google Consent Mode.

Now, briefly, the legal framework.

I do not want to go too deeply into this here, but in short: Google Analytics is a tracking tool. That is very clear. Under ePrivacy and the TTDSG, it requires consent.

This is intensified by the connection to Google Ads and remarketing. I would definitely recommend checking the Analytics settings and making sure those links are disabled where appropriate. By default, that setting in Analytics is often enabled. Analytics can automatically place a Google Ads or Google remarketing cookie and send data to Google remarketing. So it is not only analytics, but immediately also marketing.

Therefore, I recommend deactivating that connection.

Analytics is generally legally sensitive because it creates profiles and partly enriches data with user data such as age, gender, and similar information. For that reason, it is at least legally questionable.

There may be cases and ways to argue that Analytics does not require consent, but that is becoming increasingly difficult, especially if you look at the position of the authorities and their statements. To be safe, Analytics should really only fire with consent.

Google Tag Manager is also an ongoing issue. Is it functional or not? In theory, it is purely functional. It does not itself set cookies or process data in the same way.

But there is still a third-party provider behind it. The question is whether this third-party service is a service expressly requested by the user. That is how TTDSG words it: a service expressly requested by the user does not require consent. But does the user explicitly want Google Tag Manager? Probably not.

Are there things inside Tag Manager that the user necessarily wants? In many cases also not. In many cases, Tag Manager contains tracking-related things. So one can ask why it is technically on the page at all.

If I only have Analytics inside Tag Manager, then the simple question is: why is Analytics not directly on the website? Why is Tag Manager needed in between?

So there are enough arguments against treating Tag Manager as purely functional. But there are also customers who say that Tag Manager is genuinely functional for them, and they argue that position.

To be safe, I would recommend not treating Tag Manager as functional, but rather declaring it as a tool for Analytics and obtaining consent.

Consent Mode can help a bit legally. It prevents certain cookies and switches off certain Analytics tracking mechanisms. It also makes working with Tag Manager a little easier. From a legal point of view, turning Consent Mode on does not hurt; it improves things a little. But Google remains Google, and Google remains a US provider.

That means even if Analytics tracks less through Consent Mode and does not set cookies, we still have the Schrems II issue: a US provider cannot necessarily process European data with sufficient protection. So legal problems remain.

At least regarding Tag Manager, as mentioned, there are still some technical limitations. It is not yet fully convenient. But I am confident Google is working on it. More functionality will probably come in the future, making Consent Mode in Tag Manager more comfortable.

That is the legal framework.

First question: What happens if a user uses an ad blocker and Google Tag Manager is blocked? If Google Tag Manager is blocked, consentmanager would not be displayed, correct?

Yes, exactly. If your setup is that Tag Manager is on the website and consentmanager is inside Tag Manager, and the user visits the website with an ad blocker that blocks Tag Manager, then of course everything connected to Tag Manager is gone or blocked — including consentmanager.

In the best case, if you have configured everything correctly according to our recommendations, the user is still protected. Ideally, all settings or tools should only be activated when consentmanager explicitly activates them.

This returns us to the earlier point: all codes that are not in Tag Manager — such as a YouTube video embedded directly in the website — need to be adjusted. For YouTube, for example, around 20 characters need to be added to the code. We have shown this in one of our earlier videos.

Without that adjustment, YouTube would play automatically. With the adjustment, YouTube only starts when consentmanager explicitly activates YouTube. If consentmanager is blocked, it cannot activate YouTube, so the user remains protected.

The only disadvantage from a legal point of view is that the user may not be able to inform themselves about things they cannot see. That means some information obligations may not be fulfilled. From a practical perspective, certain functionality is missing and cannot be restored. Marketing functionality may also be affected.

Next question: What happens if a vendor has an additional IAB ID? Should both IDs be checked for the trigger, or can the IAB ID be ignored?

There are some providers that have IDs for both IAB and Google, for example. The IAB providers are special in that their IDs appear in the data. Next question: The crawler reports “Google General.” What is the provider Google General and how do I configure it?

Very good question. This comes up often.

It is not only Google; other providers have the same issue. Some providers have many products. Google has many products: Google Maps, reCAPTCHA, Translate, Analytics, Tag Manager, and so on. Amazon has several products. Microsoft has several products.

So we decided that it does not make sense to have Google as just one provider and then block everything together. Sometimes you want to differentiate. For example, Google reCAPTCHA may be functional, Translate may also be functional, Maps may be marketing, Analytics may be analytics, and so on.

That is why there are multiple Google providers, or multiple providers with the same parent name.

In addition, Google has many things that cannot clearly be assigned to one specific product. For example, Google Fonts or various Google APIs. There are many things used by several Google tools. reCAPTCHA, for example, may use font domains as well. Some domains are not specific to a single Google service.

That is why Google General exists. It collects everything that does not belong clearly to one of the other services.

How should you configure it in your provider list? Look at which other Google services you use. If you use Analytics, then Google General probably belongs in the Analytics area. If you use Google Ads, then it probably belongs there. If only Google General appears in your provider list because the crawler found it, then the reason may be that you use Google Fonts, for example.

In that case, you need to look carefully at why Google appeared. You can also create your own providers to classify things more precisely. But in most cases, it is possible to distinguish reasonably well.

Also worth noting: fonts are probably not essential in most cases. Many Google products are probably not essential, because you could say: why must the font be loaded from Google? Download it and host it on your own server. Then there is no third-party provider involved, and the data protection issue becomes much less severe.

Next question: Do you need the user’s consent if Google Tag Manager is used on the website because data is transferred to Google in the USA? Would Google Tag Manager be a technically necessary service?

We touched on this earlier. With Tag Manager, you can certainly provide an argument. Authorities will not simply say yes or no without listening to your reasoning. They do consider how something is justified.

But with some services it is much harder to find a good justification. Tag Manager can be such a case, because many things can work without Tag Manager, and many things inside Tag Manager are not things expressly requested by the user.

Then there is the issue of the USA and Google. So, if you want to be legally safe, you should also make Tag Manager subject to consent.

If you do not want to do that, definitely speak to a lawyer and possibly also your data protection officer. Use the options available to you to reduce risk.

Next question: Which implementation or trigger variant do you recommend in order to measure as many page views as possible?

That is difficult. The question is whether you are asking from a legal perspective, or whether the goal is simply to measure as much as possible.

If it has to be Google Analytics, the recommendation would basically be not to put Google Analytics into Tag Manager, if possible, to avoid having an additional tag in between. Place Analytics directly in the website, load consentmanager before Analytics, and definitely pay attention to the sequence. Enable Consent Mode so that Analytics tracks as little as possible before consent.

That at least gives you a somewhat more protected setup.

If Analytics must be placed in Tag Manager, then because of the current limitations of Consent Mode, it is often more sensible to build trigger groups — for example, page view plus Analytics has consent — rather than relying only on Consent Mode at that point.

But if the question is really about measuring as many page views as possible, then direct implementation in the website is usually best.

Next question: What is the difference between Page View and Consent Initialization?

In Tag Manager, the typical built-in events are Page View, Consent Initialization, and Initialization. Page View is simply when the page has loaded or is loading. Consent Initialization comes before Page View. Initialization is even earlier.

The order is: Initialization, Consent Initialization, Page View.

The purpose is that when the Community Template is used, the default status and certain settings can be set during the Consent Initialization phase before everything else starts. If I were to tell consentmanager to load on Page View, it would not be completely clear in which order tags load. It could happen that Analytics fires first and then consentmanager loads, which means Analytics would not yet have the information.

So it makes sense to use Consent Initialization, so that consentmanager is clearly before Page View.

Next question: Do I not also need to include the condition “cmpGDPR equals 1” in the trigger?

In principle, yes.

As shown earlier, there are several variables: purposes, vendors, GDPR, and a few more. In most cases, it is enough to check only the provider, because for most customers the setting is that GDPR applies to all users.

That is especially the case if you are a European company. As a European company, you must apply the GDPR to all users, regardless of where they come from. If you are a non-European company, you could change the CMP settings so that, for example, only EU visitors are asked.

In the CMP settings there is an option under GDPR settings: “Ask all visitors for consent.” That is the default. Or you could choose only EU visitors.

Only if you use such a setting does it make sense to add a second condition: cmpGDPR equals 1. In that case, Analytics would only fire if the provider has consent and GDPR applies.

However, you would then also need a second trigger for the case where cmpGDPR equals 0. If GDPR does not apply, then the layer would not be shown. An American user, for example, would come to the website, the layer would not be displayed, and the user could not click Accept. So if GDPR does not apply, Analytics could fire immediately. That would require a separate trigger.

For most customers, especially European customers, the default setting is that all visitors are asked. In that case, this second branch is not necessary.

Next question: Where do I insert the code? Does it also work via the WordPress plugin?

The WordPress plugin has become quite extensive and now has many features. If you have a WordPress website, it is worth looking at the plugin.

If you use the WordPress plugin, you usually do not need the Community Template anymore. You do not need to deliver consentmanager through Tag Manager. The recommendation would actually be to deliver consentmanager via the WordPress plugin and configure everything else through Tag Manager.

The WordPress plugin itself can also adjust code on the website. For example, with the YouTube video example from earlier, the WordPress plugin can automatically adjust YouTube videos on your website, depending on the settings you choose.

So if you use WordPress, the recommendation is usually to use the plugin rather than the Community Template. But there are of course cases where the reverse makes more sense.

If you do not use WordPress, there are also plugins for other shop systems and CMS systems. Unfortunately, some of those are not as comfortable as the WordPress plugin. In those cases, it may make more sense to work through Tag Manager.

Next question: If I have to include consentmanager in Tag Manager, do I need to classify Tag Manager as functional, otherwise the cookie banner will not be displayed?

If Tag Manager is on the website and consentmanager is inside Tag Manager, then it is basically irrelevant how you define Tag Manager inside consentmanager, because the sequence already determines the logic.

Tag Manager is already loaded before consentmanager can load. That means any blocking that might happen would already be too late, because Tag Manager is already there.

In that case, it basically only makes sense to mark Tag Manager as functional. All other settings would only confuse users. However, as mentioned earlier, this is not our recommended setup from a legal perspective. But if you do it that way, then yes, it only makes sense to mark Tag Manager as functional.
